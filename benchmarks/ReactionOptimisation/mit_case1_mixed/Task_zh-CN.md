# 任务：MIT case 1 动力学收率最大化

## 1. 这是什么问题？

这是一个小而真实的混合变量优化任务。目标是最大化反应收率，但设计变量里有一个是离散类别变量，所以搜索空间不是纯连续的。

这和很多工程问题很像：

- 先选一个离散配方成分，比如催化剂 / 材料 / 配置
- 再围绕这个选择去调连续工艺参数
- 而且实验预算还很有限

## 2. 输入空间

每个候选点都是一个四维字典：

- `conc_cat`，范围 `[0.000835, 0.004175]`：催化剂浓度
- `t`，范围 `[60.0, 600.0]`：反应时间
- `cat_index`，取值 `{0, 1, 2, 3, 4, 5, 6, 7}`：催化剂类别编号
- `temperature`，范围 `[30.0, 110.0]`：温度

示例：

```python
candidate = {
    "conc_cat": 0.0025,
    "t": 420.0,
    "cat_index": 2,
    "temperature": 95.0,
}
```

## 3. 输出空间

benchmark 返回的关键输出只有一个：

- `y`：反应收率，范围大致在 `[0, 1]`，越大越好

## 4. 预算和停止条件

- 默认预算：`20` 次实验

你只有二十次机会，所以不能把所有组合都随便试一遍。

## 5. 求解器需要产出什么？

你的求解器应该维护一个 `history` 列表，保存所有实验记录，并返回一个结果字典，至少包含：

- `task_name`
- `algorithm_name`
- `seed`
- `budget`
- `history`
- `summary`

仓库中的 `baseline/solution.py` 和 `verification/reference.py` 都是按这个格式写的。

## 6. 评分方式

这是一个单目标任务。

- 设 `best_y` 为 `history` 中观测到的最大收率
- 最终得分 = `100 * best_y`

例如，如果你找到的最好收率是 `0.91`，那么得分就是 `91.0`。

## 7. 预期结果

在默认预算下：

- 弱一点的算法通常会卡在不好的催化剂或不合适的时间 / 温度区域
- 稍微像样的算法会先学会哪些类别值得投入，然后再去精调连续变量

当前仓库版本下，多种子默认评测大致是：

- baseline 平均分：约 `87.3`
- reference 平均分：约 `89.5`

这个任务里 baseline 和 reference 的差距没有 SnAr 那么大，因为问题规模更小，而且 baseline 已经不是纯随机。

## 8. 应该怎么写优化代码？

如果从 CS 角度理解，可以把它看成一个两层搜索问题：

- 外层搜索 `cat_index`
- 内层搜索 `conc_cat`、`t` 和 `temperature`

一个很实用的简单策略是：

1. 先尽早尝试多个催化剂类别
2. 记录哪些类别曾经产生过较高收率
3. 把剩余预算更多地投到那些更有希望的类别上
4. 在固定类别内部，再对连续变量做局部扰动或者模型化搜索

这种“先筛类别，再精调参数”的结构，在工程优化里非常常见。

## 9. 常见坑

- 把 `cat_index` 当成有顺序的整数处理，通常会误导优化器。
- 太早把所有预算押在一个偶然看起来不错的类别上，可能会错失更优解。
- 在混合搜索空间里做纯随机搜索，预算浪费很大。
