# 任务：连续流 SnAr 反应优化

## 1. 这是什么问题？

这个任务模拟的是一个连续流反应优化闭环。你选择一组反应条件，调用黑盒实验，观察两个输出，然后在固定实验预算内不断重复。

这里的工程权衡是：

- 提高 `sty`，也就是更高的生产率 / 吞吐能力
- 降低 `e_factor`，也就是减少单位产品对应的废物生成

在真实工艺开发里，这个权衡很有价值：产能高但废物太多的工艺通常不可接受，而很“绿色”但产能太低的工艺也往往没有经济性。

## 2. 输入空间

每次实验都是一个包含四个连续变量的 Python 字典：

- `tau`，范围 `[0.5, 2.0]`：停留时间
- `equiv_pldn`，范围 `[1.0, 5.0]`：试剂当量比
- `conc_dfnb`，范围 `[0.1, 0.5]`：进料浓度
- `temperature`，范围 `[30.0, 120.0]`：反应温度

示例：

```python
candidate = {
    "tau": 1.2,
    "equiv_pldn": 2.5,
    "conc_dfnb": 0.35,
    "temperature": 85.0,
}
```

## 3. 输出空间

benchmark 会返回一条记录，里面包含输入和测得输出。真正参与评分的是两个输出：

- `sty`：越大越好
- `e_factor`：越小越好

## 4. 预算和停止条件

- 默认预算：`24` 次实验
- 到达预算上限就必须停止

预算故意设置得很小，目的是考查“样本效率”，不是暴力穷举。

## 5. 你的求解器需要产出什么？

从代码角度看，你的求解器应该：

1. 用 `task.create_benchmark()` 创建环境。
2. 不断提出合法候选点。
3. 调用 `task.evaluate(experiment, candidate)`。
4. 把每次返回的记录保存到 `history`。
5. 返回或打印一个包含 `history` 和 `task.summarize(history)` 的结果字典。

当前提供的 baseline 和 reference 都是按这个接口写的。

## 6. 评分方式

这是一个 Pareto 优化任务，不能只看单个“最好点”。

评分基于你观测到的折中解集合的超体积：

- `sty` 用 `13000` 做归一化
- `e_factor` 会先变成“越大越好”的生态得分：`1 - e_factor / 500`
- 两个归一化值都会截断到 `[0, 1]`
- 再在变换后的最小化空间里，相对参考点 `(1, 1)` 计算二维 hypervolume
- 最终得分 = `100 * hypervolume`

直观理解：

- 分数越高，说明你找到的折中解集合越好
- 只把一个目标优化到极致通常不够
- 同时覆盖“高产能”和“低废物”区域的算法会得分更高

## 7. 预期结果

在默认预算下：

- 弱一些或者纯随机的算法，通常只能拿到中等偏低分
- 仓库里的 baseline 代表一个简单下限
- SUMMIT 参考解明显更强

当前仓库版本里，多种子默认评测大致是：

- baseline 平均分：约 `57.5`
- reference 平均分：约 `86.1`

你不需要一定达到参考解，但如果你的算法长期低于 baseline，通常说明搜索策略还需要改进。

## 8. 该怎么写优化代码？

如果你有 CS 背景，可以把它理解成一个序贯决策问题：

- 状态：目前已经观察到的全部实验记录
- 动作：选择下一组实验条件
- 环境：`task.evaluate(...)`
- 奖励代理：多目标分数，或者临时构造的标量化目标

一个很实用的起步方案是：

1. 先采样几组比较分散的初始点
2. 用归一化后的加权和临时把双目标变成单目标
3. 围绕当前表现好的点做局部扰动
4. 偶尔跳回随机点做探索
5. 保留一组好的折中解，而不是只保留一个 incumbent

## 9. 常见坑

- 只优化 `sty`，很容易把 `e_factor` 做坏。
- 只优化 `e_factor`，可能得到经济性很差的工艺。
- 只保留一个最优点，会丢失 Pareto 多样性。
- 不做归一化时，一个目标可能会在数值上压倒另一个目标。

## 10. 本任务目录中的文件

- `task.py`：正式任务定义
- `baseline/solution.py`：不依赖 SUMMIT 优化器的简单基线解
- `verification/reference.py`：使用 SUMMIT 的较强参考解
- `verification/evaluate.py`：统一运行并输出评分结果
