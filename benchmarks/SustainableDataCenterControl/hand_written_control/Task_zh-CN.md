# 任务说明：为 SustainDC 编写手工控制函数

## 1. 任务目标

在不训练模型的前提下，为 SustainDC 的三个原始 agent 编写控制函数。

你的函数接收环境给出的三个 agent 观测向量，并返回三个离散动作：

- `agent_ls`：负载迁移
- `agent_dc`：数据中心冷却控制
- `agent_bat`：电池充放电控制

你主要需要修改的文件是：

`baseline/solution.py`

必须保留的入口函数是：

```python
def decide_actions(observations) -> dict:
    ...
```

可选函数：

```python
def reset_policy() -> None:
    ...
```

如果你希望策略具有“记忆”，可以把状态存在模块级变量里，并在 `reset_policy()` 里清空。

## 2. 你到底在解什么问题？

SustainDC 模拟了一个数据中心，每 `15` 分钟要做三类耦合决策：

1. 某些可延迟计算任务是现在执行，还是往后排？
2. 冷却设定点要降低、保持不变，还是放宽？
3. 电池要充电、放电，还是保持空闲？

这三件事互相影响：

- 延迟任务会改变未来服务器负载
- 服务器负载会改变冷却需求
- IT 负载和冷却负载共同决定电网取电
- 电网取电和电池动作共同决定碳排放

所以这不是三个互不相关的小题，而是一个被拆成三个 agent 的联合控制问题。

## 3. 你要优化什么

这个 benchmark 主要鼓励你降低：

- 碳排放
- 水耗

同时避免任务处理质量变差：

- 任务被丢弃
- 任务拖得太久而过期

固定参考策略是一个 `noop` 控制器：

- `agent_ls = 1`
- `agent_dc = 1`
- `agent_bat = 2`

你的分数衡量的是：在固定评测场景上，你比这个参考控制器好多少。

## 4. 函数输入与输出

## 输入

`decide_actions(observations)` 的输入是一个字典：

```python
{
    "agent_ls": np.ndarray(shape=(26,)),
    "agent_dc": np.ndarray(shape=(14,)),
    "agent_bat": np.ndarray(shape=(13,)),
}
```

这些数组就是 `sustaindc_env.py` 真正生成出来的原始观测向量。

## 输出

你需要返回：

```python
{
    "agent_ls": int,
    "agent_dc": int,
    "agent_bat": int,
}
```

每个值都必须是对应 agent 的合法离散动作。

## 5. 动作含义

## `agent_ls` 动作

- `0`：把可延迟任务放入队列，往后推
- `1`：保持当前队列策略不变
- `2`：从队列中取任务执行

## `agent_dc` 动作

- `0`：降低冷却设定点，意味着更强冷却
- `1`：冷却设定点不变
- `2`：提高冷却设定点，意味着更弱冷却

## `agent_bat` 动作

- `0`：充电
- `1`：放电
- `2`：空闲

## 7. 固定评测场景

benchmark 使用四个固定场景：

- `az_july`：亚利桑那夏季
- `ca_april`：加州春季
- `ny_january`：纽约冬季
- `tx_august`：德州夏末

每个场景持续 `2` 天，每天 `96` 个控制步，所以一次完整评测总共包含：

- `4` 个场景
- 每个场景 `192` 步
- 总共 `768` 步

月份下标沿用仓库原始约定：

- `0 = 1 月`
- `11 = 12 月`

## 8. 评测时统计的指标

每个 episode 会累计：

- `carbon_kg`：总碳排放
- `water_l`：总水耗
- `dropped_tasks`：总丢弃任务数
- `overdue_tasks`：累计过期任务惩罚
- `grid_energy_kwh`：总电网能耗

分数直接使用前四项；`grid_energy_kwh` 也会输出，方便你调试策略。

## 9. 评分方式

分数会把你的策略和同一场景下的 `noop` 参考策略进行比较。

先计算相对改进：

```text
carbon_gain = max(0, 1 - candidate_carbon / noop_carbon)
water_gain  = max(0, 1 - candidate_water  / noop_water)
```

为了让重复评测更稳定，小于 `0.2%` 的改进会被视为数值噪声，直接按 `0` 处理。

再计算加权改进比例：

```text
improvement_fraction = 0.85 * carbon_gain + 0.15 * water_gain
```

最后转成分数：

```text
base_score = 100 * sqrt(improvement_fraction)
safety_penalty = 5 * dropped_tasks + 0.5 * overdue_tasks
final_score = clip(base_score - safety_penalty, 0, 100)
```

### 怎么理解这个分数

- `0`：没有优于 `noop`
- 正分：说明你在不明显破坏任务质量的前提下改善了资源效率
- `100`：评分规则的数学上限

重要说明：

- `100` 不是这个复杂环境的已知全局最优策略值
- 它只是当前评分函数的严格上限

## 10. 预期结果

当前提供的 baseline 大约能得到：

`7.9 / 100`

不同次运行之间，模拟器可能带来少量波动。

这是一个故意做得比较弱、比较容易理解的起点，不是一个很强的目标结果。

## 11. 推荐的优化路径

如果你刚接触这个任务，最推荐这样入手：

1. 先优化 `agent_ls`
2. 初期让 `agent_dc` 保守
3. 在真正理解充电什么时候有收益之前，让 `agent_bat` 保持空闲

原因：

- 负载迁移本身就能带来可测的收益
- 电池策略很容易因为“先充电”而把当前碳排拉高
- 冷却策略虽然能降水耗，但用不好也可能在别的地方把能耗拉回去

## 12. 实战写代码建议

- 先从阈值规则和 `if/else` 开始。
- 每次运行后看 `verification/last_eval.json`。
- 不要只盯着最终分数，也要看原始指标。
- 如果分数突然掉到很低，先检查 `dropped_tasks` 和 `overdue_tasks` 是否变正。
- 如果碳排更高了，通常是电池或冷却策略太激进。

## 13. 最小合法示例

```python
def decide_actions(observations):
    return {
        "agent_ls": 1,
        "agent_dc": 1,
        "agent_bat": 2,
    }
```

这是一个合法策略，但它只会等价于 `noop`，所以分数大约会是 `0`。

## 14. 如何运行评测

```bash
python verification/evaluate.py
```

可选：

```bash
python verification/evaluate.py \
  --solution path/to/your_solution.py
```

如果你的上游 `dc-rl` clone 不在同级 `sustaindc/` 目录下，还可以显式指定：

```bash
python verification/evaluate.py \
  --sustaindc-root /absolute/path/to/dc-rl
```

## 15. 什么样的提交算是成功

一个好的提交通常具有这些特点：

- 代码清晰、容易读
- 能有意识地利用观测特征
- 相比 `noop`，能降低碳排和/或水耗
- 不会导致任务被丢弃或长期积压

你完全不需要机器学习，依然可以在这个 benchmark 上写出有价值的优化策略。
